00 / 04

The lifecycle

What “a magic link” actually is — from the moment Mariana clicks Send to the moment Daniel’s signature is affixed. The mechanics are the same across all three variations.
01 · Mariana Greenroom · deal entry
Clicks Save & re-send.
Form is complete; rail says all required fields filled. Action upserts the deal and creates a token.
token + email
02 · Inbox Daniel · WME
Email arrives within seconds.
Subject names the artist + date. One big Review and sign button. The structured deal is rendered inline as a preview, so opening the email doesn’t require a click.
click → landing
03 · Magic link /sign/… no login
Daniel sees the structured recap.
Lands on a clean branded page. Opening the link doesn’t count as signing — only the explicit button click does. Sign · Decline · Ignore.
Token
32-char URL-safe string. Stored with role (sign or view), recipient email, deal version, expiry timestamp.
Expiry
14 days by default. After that the link returns a friendly “ask Mariana to re-send” page.
Use
Sign links are single-use for sign/decline actions. View links are unlimited reads. Either can be visited many times without committing.
Amendments
If the deal is edited after signing, the old link is invalidated and a fresh one goes out. The agent always reviews the latest version.
A / 04

No modal, no confirmation gate. Click Save & re-send; the rail itself transforms into a confirmation, then a status panel. Best for Mariana’s 8–15-deals-a-week reality.

Why this works. Captures every real-world stakeholder — the signing agent, the road-side TM, the cc’d label rep — with role-aware magic links. Pre-empts every “can you loop in…” reply before it happens. Sets up a future where roles compose across slices (settlements, advances, etc.). Why it might not. For 80% of deals it’s overbuilt. Mariana will end up clicking through the composer with the same single recipient every time, which is slower than variation A’s one click.
04 / 04

Compare

Where each variation earns its keep.
A · Inline & immediate
Rail rewrites itself.
single-recipient
Best for the steady-state — Mariana’s 8–15 deals a week. One click sends, the rail switches to status, and she keeps moving.

Risk: typo in the agent’s email goes silently. Needs a read-receipt and a clear “bounce” surface to be safe.
B · Pre-flight checklist
One step of confirmation.
explicit
Best for the first 50 deals — the “watch Mariana enter 50 real deals manually” phase from the spec. Guardrails are visible and the TM checklist gives Diego a real artifact to use at the back office.

Risk: two-click sends accumulate friction. Needs an opt-out once Mariana has muscle memory.
C · Multi-recipient compose
Roles for everyone.
future-proof
Best for deals where more than one party needs visibility — agent + TM + label, opener-agent + co-headliner agent, manager + lawyer. Sets up the role primitive for later slices (settlement, advancing).

Risk: overkill for solo deals. Needs a “skip composer” shortcut so the 80% case stays fast.
Recommendation
Ship A · Inline & immediate as the default flow — it’s the lightest path from a finished deal to a sent link, and the data-send-button from slice 02 is already gated, so the “am I ready?” question is already answered before send. Borrow B’s disclosure list as a one-time onboarding tooltip the very first time a venue sends, so the 14-day · single-use · amendments invalidate rules are visible once and remembered. Keep C’s composer behind a small + more recipients link that quietly upgrades the inline flow when needed — the role primitive is too useful to drop, but doesn’t need to be the default.

The tour manager view is the same across all three — differences are in what extras live below the recap (settlement checklist in B, venue contacts in C). Start with the recap alone; add the rest after watching tour managers actually open the link.
